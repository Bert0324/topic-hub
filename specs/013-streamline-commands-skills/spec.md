# Feature Specification: Streamline Commands & Skills

**Feature Branch**: `spec/013-streamline-commands-skills`  
**Created**: 2026-04-11  
**Status**: Draft  
**Input**: User description: "1. IM 里的 /topichub 前缀不需要，@bot 就是类似的意思，直接用 /create 这种。2. CLI 里的身份管理只有超管才有权限，普通用户可以查看自己的身份详情。3. Skill Repo 管理全部去掉，不需要 skill 脚手架；skill 概念与业内通用。4. 不需要安装、启用、禁用、运行、卸载、setup 等 skill 生命周期概念，remote 发布的 skill 所有人都可以主动使用；仅保留 create 和 publish。5. 群组管理应该是创建 topic，概念保持一致。6. 监控指令全去掉。"  
**Depends on**: spec 011 (Superadmin Identity Model), spec 012 (Unified Skill Center)

## Clarifications

### Session 2026-04-11

- Q: Should skill-repo commands be retained, and if so which subcommands? → A: Remove all skill-repo commands entirely. Only retain `publish <path>` as a standalone command that accepts a skill directory path or address.
- Q: Should the old `/topichub` prefix be supported for backward compatibility? → A: No. Remove all backward compatibility. The `/topichub` prefix is no longer recognized; commands use short-form only.
- Q: What additional security measures should protect the IM→local execution chain beyond identity/executor token validation (spec 011)? → A: Token chain only. Spec 011's identity token + executor sub-token + IM binding via `/register` is sufficient. No additional rate limiting, command allowlists, or interactive confirmation at the command layer.
- Q: When a bound executor is at max capacity, should the system queue, reject, or auto-route to another executor? → A: Reject immediately. No auto-routing — IM has no visibility into unpaired executors; only the explicitly `/register`-bound executor is known. The user must wait or switch binding manually.
- Q: Should `publish <path>` accept remote URLs (e.g., git repos) in addition to local paths? → A: Local path only. The user must have the skill files on their local machine to publish.
- Q: What is the pairing flow direction between executor and IM accounts? → A: The local executor generates a pairing code (displayed in CLI). The IM user enters `/register <code>` to bind their IM account to that executor. The relationship is 1 executor : N IM accounts — one executor can serve commands from multiple IM platforms/accounts. This reverses the previous flow where IM generated the code and CLI linked.
- Q: Can a single IM account be bound to multiple executors simultaneously? → A: No. Each IM account is bound to exactly one executor at a time. `/register` replaces any previous binding.
- Q: With pairing reversed (executor generates code, IM registers), should CLI `link`/`unlink` commands be retained? → A: Remove both. Pairing is fully managed via IM `/register` and `/unregister`. No CLI-side pairing step remains.
- Q: Which IM commands should work without requiring executor binding? → A: `/help` and `/register` do not require binding. `/help` returns fixed content (command list); `/register` is the binding mechanism itself. All other commands require an active executor binding.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Simplified IM Commands Without Prefix (Priority: P1)

A user @-mentions the bot in an IM channel and issues commands directly — `/create`, `/update`, `/show`, etc. — without a `/topichub` prefix. The @-mention to the bot already establishes the command context, making the prefix redundant. The system recognizes these short-form commands and processes them identically to the old `/topichub <action>` form. The `/answer` command also works without prefix. The `/register <code>` command accepts a pairing code generated by a local executor to bind the IM account to that executor (1 executor : N IM accounts).

**Why this priority**: This directly improves the IM user experience for every single interaction. The `/topichub` prefix adds friction with no value when the bot is already explicitly addressed.

**Independent Test**: Can be tested by @-mentioning the bot and issuing `/create bug`, then verifying the system creates a topic just as `/topichub create bug` did before.

**Acceptance Scenarios**:

1. **Given** a user @-mentions the bot in an IM channel, **When** the user sends `/create <type> <args>`, **Then** the system parses and processes it as a topic creation command.
2. **Given** a user @-mentions the bot (even without an executor binding), **When** the user sends `/help`, **Then** the system returns the list of available commands using the new short-form syntax — no binding required.
3. **Given** a user @-mentions the bot, **When** the user sends `/show`, **Then** the system shows the current active topic details in the channel.
4. **Given** a user sends a command without @-mentioning the bot, **When** the system receives the message, **Then** it ignores the message (no accidental command interpretation).
5. **Given** a user @-mentions the bot, **When** the user sends `/use <skill-name> <args>`, **Then** the system dispatches the skill invocation to the user's bound executor.

### User Story 2 - Superadmin-Only Identity Management in CLI (Priority: P1)

Identity management commands (create, list all, revoke, regenerate-token) are restricted to superadmin-authenticated CLI sessions. A regular user running the CLI can view their own identity details (display name, unique ID, status, active executors) but cannot create, revoke, or modify other identities. The CLI validates the caller's token and enforces this boundary before executing any identity subcommand.

**Why this priority**: Proper access control for identity management is essential for system security. Without it, any token holder could create or revoke identities, which undermines the superadmin model from spec 011.

**Independent Test**: Can be tested by running `identity create` with a regular identity token and verifying it is rejected, then running `identity me` with the same token and verifying it returns the caller's own details.

**Acceptance Scenarios**:

1. **Given** a regular user with a valid identity token, **When** they run `identity create --unique-id x --name y`, **Then** the system rejects the request with a "superadmin only" error.
2. **Given** a regular user with a valid identity token, **When** they run `identity me`, **Then** the system returns their own display name, unique ID, status, and active executor count.
3. **Given** the superadmin, **When** they run `identity create`, `identity list`, `identity revoke`, or `identity regenerate-token`, **Then** the system executes the command successfully.
4. **Given** a regular user, **When** they attempt `identity list` (listing all identities), **Then** the system rejects the request with a "superadmin only" error.
5. **Given** a regular user, **When** they attempt `identity revoke --id <any-id>`, **Then** the system rejects the request.

---

### User Story 3 - Simplified Skill Lifecycle: Create and Publish Only (Priority: P1)

The entire skill lifecycle is reduced to two operations: **create** (local authoring) and **publish** (sharing to Skill Center). There is no install, enable, disable, setup, uninstall, or config step. Any skill published to the Skill Center is immediately available to all users — they simply invoke it by name from CLI or IM, and the system automatically pulls and caches the skill definition locally. The skill-repo scaffolding and its associated commands are removed entirely.

**Why this priority**: The existing multi-step skill lifecycle (install → enable → setup → config) creates unnecessary friction. Since skills are just SKILL.md files executed locally, the only meaningful transitions are creation and sharing.

**Independent Test**: Can be tested by creating a skill with `skill create`, publishing it with `publish`, then having another user invoke it by name and verifying it executes — with no install/enable/setup steps in between.

**Acceptance Scenarios**:

1. **Given** a user wants to create a new skill, **When** they run the `skill create` CLI command, **Then** the system scaffolds a minimal SKILL.md-based skill in the local skills directory.
2. **Given** a user has a local skill, **When** they run the `publish` CLI command for that skill, **Then** the skill is published to the Skill Center and becomes available to all users.
3. **Given** a published skill exists in the Skill Center, **When** any user invokes it by name (CLI or IM), **Then** the system automatically pulls it to local and executes it — no install/enable step required.
4. **Given** the updated system, **When** a user runs `skill install`, `skill enable`, `skill disable`, `skill setup`, `skill config`, or `skill uninstall`, **Then** the system responds with "unknown command" or helpful guidance pointing to the new workflow.
5. **Given** the updated system, **When** a user runs any `skill-repo` subcommand, **Then** the system responds with "unknown command."

### User Story 4 - Topic Creation Replaces Group Management (Priority: P2)

The `group create` CLI command is replaced by `topic create`, aligning the CLI vocabulary with the core domain concept. Creating a topic from the CLI follows the same semantics as creating one from IM — it associates a topic with a platform and channel, assigns a type, and triggers the standard topic creation pipeline. The "group" concept as a separate management unit is removed.

**Why this priority**: Aligning CLI terminology with the domain model eliminates confusion. Users think in terms of topics, not groups. However, this is primarily a naming/routing change rather than new functionality.

**Independent Test**: Can be tested by running `topic create <type> --platform <p> --channel <c>` from the CLI and verifying a topic is created, then confirming `group create` no longer works.

**Acceptance Scenarios**:

1. **Given** a superadmin or authorized user, **When** they run `topic create <type> --platform <p> --channel <c>`, **Then** a new topic is created in the specified channel.
2. **Given** the updated system, **When** a user runs `group create`, **Then** the system responds with "unknown command" and suggests using `topic create` instead.
3. **Given** a topic created via CLI, **When** viewed from IM, **Then** it appears and behaves identically to a topic created from IM.

---

### User Story 5 - Removal of Monitoring Commands (Priority: P3)

The `health` and `stats` CLI commands are removed from the user-facing CLI. System monitoring information (connected IM platforms, registered users, active executors, skill statistics) is consolidated into the superadmin dashboard in the Skill Center web UI (as defined in spec 012). The server's `/health` API endpoint remains for infrastructure monitoring tools but is no longer exposed as a CLI command.

**Why this priority**: Monitoring data is better served through the Skill Center web UI dashboard where it can be displayed visually and updated in real time, rather than through one-shot CLI commands that show a snapshot.

**Independent Test**: Can be tested by verifying `health` and `stats` CLI commands no longer exist, and that the superadmin dashboard in the Skill Center web UI shows equivalent or better information.

**Acceptance Scenarios**:

1. **Given** the updated system, **When** a user runs `health` or `stats` from the CLI, **Then** the system responds with "unknown command."
2. **Given** the superadmin opens the Skill Center web UI, **When** the dashboard loads, **Then** it displays system health, statistics, and operational metrics previously available through CLI commands.

---

### User Story 6 - Complete IM Command Set with New Syntax (Priority: P1)

The full set of IM commands is rationalized under the new prefix-free syntax. The complete command list accessible from IM (after @-mentioning the bot) is: `/create`, `/update`, `/assign`, `/show`, `/timeline`, `/reopen`, `/history`, `/search`, `/help`, `/use`, `/register`, `/unregister`, `/answer`. Each command works directly without the `/topichub` prefix.

**Why this priority**: Defining the complete IM command surface is essential for both implementation and user documentation. This story enumerates the final set to ensure nothing is missed or accidentally retained.

**Independent Test**: Can be tested by issuing each of the 13 listed commands via IM and verifying each is recognized and processed correctly.

**Acceptance Scenarios**:

1. **Given** a user @-mentions the bot, **When** they send any of the 13 listed commands with appropriate arguments, **Then** the system processes each one correctly.
2. **Given** a user @-mentions the bot, **When** they send `/topichub create`, **Then** the system treats `/topichub` as an unknown command and returns an error with guidance to use `/create` instead.
3. **Given** a user @-mentions the bot, **When** they send `/help`, **Then** the help output lists commands in the new short-form syntax (e.g., `/create`, not `/topichub create`).

---

### User Story 7 - Streamlined CLI Command Surface (Priority: P1)

The CLI (`topichub-admin`) command set is rationalized to remove redundant, deprecated, and overly complex commands. The final CLI command surface is:

- **init** — first-time setup (server URL, token, executor, skills directory)
- **serve** — start local executor
- **identity me** — view own identity (all users)
- **identity create / list / revoke / regenerate-token** — manage identities (superadmin only)
- **skill create** — scaffold a new local skill
- **publish \<path\>** — publish a local skill (by directory path or address) to the Skill Center
- **login / logout** — authenticate
- **topic create** — create a topic (replaces `group create`)

Removed commands: `auth`, `skill list/install/enable/disable/setup/config/uninstall`, `skill-repo *`, `group *`, `health`, `stats`, `link`, `unlink`.

**Why this priority**: A clean, focused CLI surface reduces cognitive load and ensures every command has a clear purpose.

**Independent Test**: Can be tested by running each retained command with `--help` and verifying it works, then running each removed command and verifying it is no longer recognized.

**Acceptance Scenarios**:

1. **Given** the updated CLI, **When** a user runs any of the retained commands, **Then** the command executes as expected.
2. **Given** the updated CLI, **When** a user runs a removed command (e.g., `skill install`, `health`, `skill-repo create`), **Then** the system displays a "command not found" message with the updated usage guide.
3. **Given** the updated CLI, **When** a user runs `topichub-admin` with no arguments, **Then** the usage output lists only the retained commands.

### Edge Cases

- What happens when an existing IM integration sends commands with the old `/topichub` prefix? The system rejects the command and returns an error with guidance to use the new short-form syntax.
- What happens when a regular user attempts superadmin-only identity commands with a crafted request bypassing the CLI? The server-side API must also enforce the superadmin check, not just the CLI.
- What happens to existing installed/enabled skills after the migration removes the install/enable lifecycle? All previously enabled skills should continue to function; their registration records are migrated to the simplified model.
- What happens when a user runs a removed command that they had in a script or alias? The error message should clearly indicate what replaced the removed command.
- What happens when `/use <skill-name>` is sent from IM but the skill does not exist in the Skill Center or locally? The system returns a clear "skill not found" message.
- What happens when a user has scripts that still use the old `/topichub` prefix? They must update immediately — no deprecation period is provided.
- What happens when a user sends a command from IM but their bound executor is at max capacity? The system rejects immediately with a "busy" message. The user can wait and retry, or `/register` to a different executor.
- What happens when multiple IM accounts are bound to an executor and it goes offline? All bound IM accounts lose their executor; commands from any of them return "executor unavailable."
- What happens when an IM user enters an expired or invalid pairing code via `/register`? The system rejects with a clear error and instructs them to get a fresh code from the executor.
- What happens when an unregistered IM user sends a command other than `/help` or `/register`? The system rejects with a message instructing them to run `/register <code>` first, and suggests `/help` to see available commands.

## Requirements *(mandatory)*

### Functional Requirements

**IM Command Simplification**

- **FR-001**: System MUST accept IM commands without the `/topichub` prefix when the bot is @-mentioned; commands are `/create`, `/update`, `/assign`, `/show`, `/timeline`, `/reopen`, `/history`, `/search`, `/help`, `/use`, `/register`, `/unregister`.
- **FR-002**: The `/answer` command MUST continue to work without prefix (no change from current behavior).
- **FR-003**: System MUST ignore messages that do not @-mention the bot, preventing accidental command interpretation.
- **FR-004**: System MUST reject the old `/topichub <action>` syntax with an error message guiding users to the new short-form commands. No backward compatibility is provided.
- **FR-005**: The `/help` command output MUST display commands in the new short-form syntax.
- **FR-028**: The `/help` and `/register` IM commands MUST work without requiring an active executor binding. `/help` returns static content; `/register` is the binding mechanism itself. All other IM commands MUST require an active binding and reject with a "register first" message if none exists.

**Identity Management Access Control**

- **FR-006**: The server API MUST enforce superadmin-only access for identity creation, listing all identities, revocation, and token regeneration.
- **FR-007**: Any user with a valid identity token MUST be able to retrieve their own identity details (display name, unique ID, status, active executor count) via a dedicated "self" endpoint.
- **FR-008**: The CLI MUST provide an `identity me` subcommand that calls the self-detail endpoint using the caller's token.
- **FR-009**: The CLI MUST reject superadmin-only identity subcommands with a clear error when invoked with a non-superadmin token, before making API calls.

**Skill Lifecycle Simplification**

- **FR-010**: The CLI MUST retain the `skill create` command for local skill scaffolding with a minimal SKILL.md-based template.
- **FR-011**: The CLI MUST retain the `publish` command, accepting a local skill directory path as argument, for publishing individual skills to the Skill Center. Remote URLs are not supported.
- **FR-012**: The CLI MUST remove the following skill subcommands: `list`, `install`, `enable`, `disable`, `setup`, `config`, `uninstall`.
- **FR-013**: The CLI MUST remove all `skill-repo` commands (`create`, `init`).
- **FR-014**: All skills published to the Skill Center MUST be immediately available to all users without any install, enable, or setup step.
- **FR-015**: When a user invokes a published skill by name, the system MUST automatically pull, cache, and register it locally for execution.

**Dispatch Behavior**

- **FR-022**: When an IM command targets a bound executor that is at maximum concurrent task capacity, the system MUST reject the command immediately and inform the user that their executor is busy.
- **FR-023**: The system MUST NOT auto-route dispatches to other executors under the same identity. Only the explicitly `/register`-bound executor receives dispatches from IM.
- **FR-024**: The local executor MUST generate a pairing code on startup and display it in the CLI console. The IM user enters `/register <code>` to bind their IM account to that executor.
- **FR-025**: The executor-to-IM-account relationship MUST be 1:N — one executor can be bound to multiple IM accounts across different platforms.
- **FR-026**: The IM `/register <code>` command MUST replace any previous executor binding for that IM account (each IM account is bound to exactly one executor at a time).

**Topic Creation (replacing Group Management)**

- **FR-016**: The CLI MUST provide a `topic create` command that creates a topic associated with a platform and channel, replacing the old `group create` command.
- **FR-017**: The CLI MUST remove the `group` command and all its subcommands.

**Monitoring Command Removal**

- **FR-018**: The CLI MUST remove the `health` and `stats` commands.
- **FR-019**: System monitoring data MUST be accessible through the Skill Center superadmin dashboard (as specified in spec 012).

**CLI Surface Rationalization**

- **FR-020**: The CLI MUST remove the `auth` command (superseded by `init` and `login`).
- **FR-021**: The final CLI command set MUST be exactly: `init`, `serve`, `identity` (with subcommands `me`, `create`, `list`, `revoke`, `regenerate-token`), `skill create`, `publish <path>`, `login`, `logout`, `topic create`.
- **FR-027**: The CLI MUST remove the `link` and `unlink` commands. IM-to-executor pairing is managed entirely via IM `/register` and `/unregister`.

### Key Entities

- **Command**: A user action issued either via IM (@bot + short-form) or CLI. IM commands no longer carry a `/topichub` prefix. CLI commands are rationalized to a minimal set.
- **Identity**: A user record with role-based access — superadmin or regular. Superadmin has full identity management capabilities; regular users can only view their own details.
- **Executor**: A local process that generates a pairing code on startup. Multiple IM accounts can bind to one executor (1:N). The executor is the origin of the pairing flow — it generates the code, and IM users consume it via `/register <code>`.
- **Pairing Code**: A short-lived, one-time code generated by a local executor and displayed in the CLI console. An IM user enters `/register <code>` to bind their IM account to that executor.
- **Skill**: A SKILL.md-based agent capability with only two lifecycle states: local (unpublished) and published. No install/enable/disable/setup/uninstall transitions.
- **Topic**: The primary domain object, now also creatable from the CLI via `topic create`, replacing the former `group create` concept.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can issue any IM command in under 20 characters of typing (excluding arguments), down from 30+ with the `/topichub` prefix.
- **SC-002**: Regular users attempting superadmin-only identity commands receive a clear rejection within 1 second.
- **SC-003**: A user can go from skill creation to invocation by another user in under 5 minutes, with only two commands (`skill create` + `publish`).
- **SC-004**: The CLI usage output lists no more than 12 top-level commands, reduced from the current 16+.
- **SC-005**: The old `/topichub` prefix is cleanly rejected with a helpful error message pointing to the new syntax.
- **SC-006**: 100% of previously functional skills remain operational after the lifecycle simplification migration.

## Assumptions

- Specs 011 (Superadmin Identity Model) and 012 (Unified Skill Center) are implemented as prerequisites. This spec builds on their identity model, skill unification, and Skill Center web UI.
- The IM bridge (OpenClaw) supports configuring the bot to respond to @-mentions with prefix-free commands — no platform-level limitation prevents this change.
- No backward compatibility is provided for the old `/topichub` prefix — it is a clean break.
- The `auth` CLI command is fully superseded by the `init` flow and `login` command; no users depend on `auth` as their sole authentication path.
- The server's `/health` endpoint remains available for infrastructure monitoring tools (e.g., load balancers, uptime checkers) even though the CLI command is removed.
- The `skill create` command produces a minimal SKILL.md template, not a full project scaffold with package.json, build tooling, or category configuration. The skill format aligns with the industry-standard concept of an agent skill (similar to Cursor superpowers).
- Topic creation from CLI serves the same use case as group creation — associating a new topic with a platform and channel. The underlying data model for topics does not change.

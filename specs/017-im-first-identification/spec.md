# Feature Specification: IM-First Identification

**Feature Branch**: `019-im-first-identification`  
**Created**: 2026-04-12  
**Status**: Draft  
**Input**: User description: "支持让用户在im里完成identification，可以让人不胡乱注册，也保障了推广的简易性；虽然多个im平台可以多次注册，但问题也不大；所有注册了的im用户都可以注册但是不能重复注册，因为权限已经做了区分"

## Clarifications

### Session 2026-04-12

- Q: Should superadmin provisioning remain, and what exact IM self-serve commands apply? → A: Superadmin-driven identity creation remains fully supported in parallel. In IM, a dedicated `/id` command exposes: (1) **`create`** — creates the participant’s user record, returns their credential token, binds to the calling IM account; the same IM user cannot complete a second successful create; **display name** is taken from the IM user’s name as provided by the integration; **unique id** is system-generated (not chosen by the user). (2) **`me`** — shows the caller’s **token**, **name**, and **id**.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Self-serve identification via IM `/id` (Priority: P1)

A new participant uses the instant-messaging interface and runs `/id create`. The system creates their participant record bound to that IM account, sets their display name from the IM profile name, assigns an opaque unique id, returns their token, and thereafter recognizes them for IM actions. They can later use `/id me` to view token, name, and id. This path does not replace elevated provisioning: operators may still create identities via the existing superadmin flow.

**Why this priority**: Primary adoption path with concrete command semantics; aligns promotion and anti-duplicate rules with explicit UX.

**Independent Test**: From an IM account with no prior registration, run `/id create` once → receive token and usable identity; run `/id me` → fields match; second `/id create` from the same IM account → blocked with a clear message.

**Acceptance Scenarios**:

1. **Given** an IM account that has not yet created a participant via `/id create`, **When** the user issues `/id create`, **Then** the system creates one participant bound to that IM account, sets **name** from the IM user’s display name, assigns a **unique id** generated by the system, returns the **token** to the user, and persists the binding.
2. **Given** a participant already created from that IM account, **When** the user issues `/id create` again, **Then** the system rejects the request and does not issue a second independent participant for the same IM account.
3. **Given** an existing participant for the caller’s IM account, **When** the user issues `/id me`, **Then** the response includes their **token**, **name**, and **id** (consistent with stored records).
4. **Given** a deployment where superadmin provisioning is used, **When** an operator creates a user through the superadmin path, **Then** that capability continues to work as before (parallel to IM self-serve).

---

### User Story 2 - Same person on a second IM platform (Priority: P2)

A participant who already ran **`/id create`** on one IM platform later uses another supported platform (different IM account). They run **`/id create`** there as well. The product allows this second registration when policy permits, so multi-channel users are not blocked.

**Why this priority**: The input explicitly accepts multiple IM platforms registering separately; this story locks that expectation into testable behavior.

**Independent Test**: Run **`/id create`** on platform A, then on platform B with a different IM account; both succeed without forcing the user to abandon the first channel.

**Acceptance Scenarios**:

1. **Given** a user who completed **`/id create`** on IM platform A, **When** they run **`/id create`** on IM platform B (different platform account), **Then** registration succeeds and both channels remain usable according to product rules.
2. **Given** two active IM registrations for different platforms, **When** routine commands run on each platform, **Then** actions are attributed consistently to the intended participant model (no silent cross-platform merge is required for v1 unless separately specified).

---

### User Story 3 - No duplicate identification for the same IM account (Priority: P1)

A participant who already used `/id create` (or an equivalent first-time bind) for a given IM account cannot create a second participant record for that same IM account.

**Why this priority**: Prevents abuse and support confusion; aligns with "不能重复注册" while still allowing multi-platform registration where platforms differ.

**Independent Test**: Run `/id create` twice from the same IM account; first succeeds, second is rejected with a clear explanation.

**Acceptance Scenarios**:

1. **Given** an IM account that has already completed `/id create`, **When** the same IM account runs `/id create` again, **Then** the system rejects duplicate creation (no second independent participant for that IM account by default).
2. **Given** two different IM accounts (including two accounts on different platforms), **When** each runs `/id create`, **Then** both succeed independently, consistent with User Story 2.

---

### Edge Cases

- Participant loses access to IM or identification is revoked: recovery must be defined (e.g., admin-assisted reset) without breaking uniqueness rules.
- IM provider sends duplicate or retried events: `/id create` must remain idempotent for the same IM account (at most one successful create).
- Participant attempts identification from a shared device or kiosk-style IM session: outcome should not weaken uniqueness guarantees for the IM account identifier.
- Elevated operators (e.g., super-administrator) remain distinguishable from standard identified users for sensitive operations.
- **`/id me` exposes a long-lived token in-channel**: acceptable for this product decision; deployments should assume users treat the IM thread as sensitive; document risk for shared-device use.
- IM display name changes after registration: behavior for whether **name** updates automatically is deferred to planning unless a later clarification requires live sync.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST retain **superadmin-driven identity creation** (unchanged in purpose): operators with elevated credentials MUST still be able to provision users without using IM self-serve.
- **FR-002**: The system MUST expose an IM command **`/id create`** that creates a participant identity **bound to the calling IM account**, sets **display name** from the IM user’s name supplied by the integration, assigns a **system-generated unique id** (not user-chosen), returns the participant’s **token** to the caller, and succeeds **at most once per IM account** for this self-serve path.
- **FR-003**: The system MUST expose an IM command **`/id me`** for participants bound via `/id create` (and, where applicable, equivalent bindings) that returns the caller’s **token**, **name**, and **id**.
- **FR-004**: The system MUST treat each supported IM account (stable account identity from the IM provider within a connected integration) as eligible for **at most one** successful **`/id create`** outcome; repeat attempts MUST yield a deterministic, participant-visible result that does not create a duplicate participant for that IM account.
- **FR-005**: The system MUST allow a participant to complete **`/id create`** on more than one IM platform when those platforms correspond to **different** IM accounts, consistent with the product’s multi-channel positioning.
- **FR-006**: The system MUST attribute IM-originated actions after identification to the correct participant model and MUST respect role separation between elevated operators and standard participants.
- **FR-007**: The IM self-serve flow MUST be unsuitable for trivial unattended bulk creation of arbitrary identities from the open internet under the product’s intended deployment (trust boundary: requests originate from configured IM integrations and provider-authenticated accounts).
- **FR-008**: The system MUST document or enforce how “already registered via `/id create`” is evaluated for an IM account so support and audits can explain why a duplicate attempt was blocked.

### Key Entities *(include if feature involves data)*

- **IM account**: A stable identifier for a user on a specific IM platform as provided by that platform to the integration (used for uniqueness and attribution).
- **Participant identity**: The product’s notion of an identified user used for permissions and history after successful onboarding; carries **unique id** (system-generated), **display name** (from IM at `/id create` for the self-serve path), and **token** (returned to the user).
- **Identification outcome**: The durable record that ties an IM account to a participant identity for first-time registration (including via `/id create`), including status needed to enforce non-duplication.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: At least 90% of participants in a controlled pilot complete first-time identification using **`/id create`** without human administrator intervention for account creation.
- **SC-002**: For the same IM account, 100% of intentional duplicate **`/id create`** attempts in test receive a clear, documented outcome (success path not offered twice).
- **SC-003**: In pilot telemetry or scripted tests, multi-platform participants can complete **`/id create`** on two distinct IM accounts within a single day without manual data repair.
- **SC-004**: Support-facing documentation explains the duplicate rule and the coexistence of **superadmin** vs **`/id`** paths in plain language; zero critical-severity tickets in pilot attributed to “unclear duplicate registration” behavior.
- **SC-005**: In usability tests, participants who run **`/id me`** after **`/id create`** successfully retrieve **token**, **name**, and **id** on first try in at least 90% of cases.

## Assumptions

- An “IM user” means an account authenticated by the IM provider and visible to the product’s integration—not an anonymous web visitor.
- “Permissions already distinguished” implies elevated roles continue to exist; this feature adds **`/id`** self-serve for standard participants **without removing** superadmin provisioning.
- **Superadmin** onboarding remains a first-class, supported path alongside **`/id create`** (not limited to break-glass).
- Linking multiple IM accounts to one logical participant profile beyond first-time registration is out of scope unless a later feature specifies merge rules; v1 may keep separate participant records per IM account if that matches implementation choices, provided duplicate same-IM **`/id create`** is forbidden.

## Out of Scope (v1)

- Removing or deprecating superadmin-driven identity creation.
- Cross-platform automatic identity merging without explicit user or admin action.
